# Webhook Trigger

## Trigger Type

`webhook`

## Use Cases
- External system callbacks (payment, messaging, notifications, etc.).
- When third-party request parameters need to be parsed for use in subsequent nodes.

## Trigger Timing / Events
- Accessing the system-generated Webhook URL (only `POST` is supported).
- Optional HTTP Basic Authentication.
- Response strategies differ between synchronous and asynchronous modes:
  - Synchronous mode: The response content is determined by the response node; if no response node is reached, the default is 200/500 based on execution status (success / failed).
  - Asynchronous mode: Returns the `response` configured in the trigger directly.

## Configuration Items
| Field | Type | Default | Required | Description |
| --- | --- | --- | --- | --- |
| basicAuthentication | false | object | false | No | HTTP Basic Authentication configuration. Fill with `false` if not enabled. |
| basicAuthentication.username | string | - | No | Authentication username (required only when enabled). |
| basicAuthentication.password | string | - | No | Authentication password (required only when enabled). |
| request | object | {} | No | Request parsing configuration. |
| request.headers | object[] | [] | No | Request header parsing configuration (used for variable option display and mapping). |
| request.headers[].key | string | - | Yes | Request header name (e.g., `x-signature`). |
| request.query | object[] | [] | No | Query parameter parsing configuration. |
| request.query[].key | string | - | Yes | Query parameter path (supports nesting, e.g., `order.id`). |
| request.query[].alias | string | - | No | Variable alias (affects display only). |
| request.query[]._var | string | - | No | Internal variable name (automatically generated by the UI, e.g., `query_$0`). |
| request.body | object[] | [] | No | JSON request body parsing configuration (only `application/json` is supported). |
| request.body[].key | string | - | Yes | JSON path (e.g., `data.id`). |
| request.body[].alias | string | - | No | Variable alias (affects display only). |
| request.body[]._var | string | - | No | Internal variable name (automatically generated by the UI, e.g., `body_$0`). |
| response | object | - | No | Response configuration for asynchronous mode; defaults to 200 with an empty body if not configured. |
| response.statusCode | number | 200 | Yes | HTTP status code. |
| response.headers | object[] | [] | No | Array of response headers, elements include `name` and `value`. |
| response.headers[].name | string | - | Yes | Response header name. |
| response.headers[].value | string | - | Yes | Response header value (supports template variables). |
| response.body | object | - | No | Response body (JSON), supports template variables. |

## Example Configuration
```json
{
  "basicAuthentication": {
    "username": "webhook",
    "password": "secret"
  },
  "request": {
    "headers": [
      { "key": "x-signature" }
    ],
    "query": [
      { "key": "event", "alias": "Event", "_var": "query_$0" }
    ],
    "body": [
      { "key": "data.id", "alias": "Order ID", "_var": "body_$0" }
    ]
  },
  "response": {
    "statusCode": 200,
    "headers": [
      { "name": "content-type", "value": "application/json" }
    ],
    "body": {
      "ok": true,
      "id": "{{ $context.body.body_$0 }}"
    }
  }
}
```

## Modeling Extracted JSON Objects

The request extraction list models only the configured item itself. If a configured header, query, or body item resolves to an object or array, its nested structure is still unmodeled.

- Extract scalar leaf paths directly when possible, for example `data.id`.
- If downstream nodes need several fields from an extracted object/array, add `json-variable-mapping` or `json-query` immediately after the trigger.
- Pass the extracted root such as `{{$context.body.body_$0}}` into the JSON node, define the required fields, and use only the JSON node's outputs afterward.
- Do not write deeper paths such as `{{$context.body.body_$0.order.id}}` directly in later nodes unless that path was explicitly modeled by the JSON node. A server-resolvable path is not enough for frontend display and editing.

## Output Variables
The variable selector for this trigger is a tree array of `{ label, value, children? }`. At runtime, join the `value` segments with `.` and prepend `$context`.

- Exposed roots are created only from the configured request extraction items: `headers`, `query`, and `body`.
- Each child node uses `alias` for display, but the runtime path uses `_var` when present; otherwise it falls back to `key`.
- That means a configured query item may display as `Event` while the actual expression is still `{{$context.query.query_$0}}`.
- An extracted scalar can be used directly. An extracted object/array must be passed through `json-variable-mapping` or `json-query` before any nested child field is used downstream.
- Example references for the sample configuration above: `{{$context.query.query_$0}}`, `{{$context.body.body_$0}}`, `{{$context.headers.x-signature}}`.
