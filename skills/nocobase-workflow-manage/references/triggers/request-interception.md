# 操作前事件

## 触发器类型

`request-interception`
请使用以上 `type` 值创建触发器，不要使用文档文件名作为 type。

## 适用场景
- 需要在数据新增、更新或删除操作执行前进行校验或逻辑判断。
- 需要根据业务规则拒绝某些操作请求（如数据校验不通过时阻止提交）。
- 希望在操作执行前对提交的数据进行预处理或审查。

## 触发时机 / 事件
- 在操作请求（通过按钮或 API 发起）实际执行**之前**触发。
- 作为中间件拦截 `create`、`update`、`updateOrCreate`、`destroy` 操作。
- **始终同步执行**（`sync = true`），流程完成后才决定是否放行请求。

## 配置项列表
| 字段 | 类型 | 默认值 | 必填 | 说明 |
| --- | --- | --- | --- | --- |
| collection | string | - | 是 | 监听的数据表，格式为 `"<dataSource>:<collection>"`（如 `"posts"` / `"mysql:orders"` 等，数据源为主数据源时可省略 `dataSource`）。 |
| global | boolean | false | 否 | 触发模式：`false` 局部模式（需通过按钮绑定），`true` 全局模式（对选定操作自动生效）。 |
| actions | string[] | - | 仅全局模式必填 | 全局模式下监听的操作类型，支持：`"create"`、`"update"`、`"destroy"`。 |

## 触发模式说明

### 局部模式（global = false）
- 需要在操作按钮上手动绑定该工作流。
- 仅当绑定的按钮触发操作时才执行此工作流。
- 适用于特定表单或按钮场景。

### 全局模式（global = true）
- 对指定数据表的选定操作类型全局生效，无需手动绑定。
- 所有匹配的操作请求都会自动触发此工作流。
- 需要通过 `actions` 字段指定监听哪些操作。

## 执行顺序
- 同一数据表上的多个工作流按顺序执行：先执行局部模式的工作流，再执行全局模式的工作流。
- 局部模式的工作流按 `triggerWorkflows` 参数中的排列顺序执行。
- 全局模式的工作流按工作流 ID 升序执行。
- 任一工作流拦截请求后，后续工作流不再执行。

## 触发器变量
- `$context.params`：请求操作的参数对象，包含以下属性：
  - `$context.params.filterByTk`：目标记录的主键值（更新和删除操作时有值）。
  - `$context.params.filter`：请求的过滤条件。
  - `$context.params.values`：请求提交的数据（新增和更新操作时有值，删除操作时无此字段）。
- `$context.user`：触发操作的用户（脱敏后的用户信息）。
- `$context.roleName`：触发操作的用户角色名称。

## 拦截机制
- 流程正常完成（状态为已完成）：请求放行，继续执行原操作。
- 流程被"结束流程"节点以失败状态终止：请求被拦截，返回 400 错误及相关提示信息。
- 流程执行异常：返回 500 错误。

## 示例配置

### 局部模式
```json
{
  "collection": "mysql:orders",
  "global": false
}
```

### 全局模式
```json
{
  "collection": "orders",
  "global": true,
  "actions": ["create", "update"]
}
```
