# CC

## Node Type

`cc`
Use the above `type` value when creating the node; do not use the documentation filename as the type.

Important: `ccUid` and `taskCardUid` are not ordinary display values. They bind `node.config` to persisted v2 FlowModel UI trees. `ccUid` points to the CC detail popup configured from **User interface**, while `taskCardUid` points to the card body shown for each item in the To-Do Center's CC list. Saving a UID string without creating the matching FlowModel tree does not create a usable custom UI.

## Node Description
Sends a CC task/notification to specified users, used for viewing or informing only.

## Business Scenario Examples
Sending a CC to relevant stakeholders for information after an approval is completed.

## Configuration Items
| Field | Type | Default | Required | Description |
| --- | --- | --- | --- | --- |
| users | array | [] | Yes | List of recipients. Specified as an array of user IDs or user queries. User IDs could be found by query `users:list` API, or use variables of user IDs form upstream. The query object will contains a `filter` object to describe the query condition of users collection.  See [Common Conventions - filter](../conventions/index.md#the-filter-field-in-trigger-and-node-configuration). |
| ccUid | string | None | Yes | Root UID of the v2 CC detail popup FlowModel, stored at `node.config.ccUid`. It must point to a `CCChildPageModel` tree created from the node's **User interface** configuration. This surface controls the blocks and content shown after a recipient opens a CC item; it is separate from the list card. Do not use the legacy `ccDetail` field as a substitute. |
| taskCardUid | string | None | No | Root UID of the v2 CC list-card FlowModel, stored at `node.config.taskCardUid`. It must point to a `CCTaskCardDetailsModel` tree created from the node's **Task card** configuration. This surface controls the fields, title, description, and layout rendered directly in each CC item in the To-Do Center; it does not control the detail popup. When omitted, the built-in fallback card is used. |
| title | string | Node Title | No | Task title, supports variable templates. |
| notifications | array | [] | No | Notification configurations (generated by UI). |

## UID-backed UI Configuration

Treat each UID as a binding from the CC node to an existing FlowModel root, not as a complete UI configuration by itself:

| Config field | Runtime surface | Expected root FlowModel |
| --- | --- | --- |
| `node.config.ccUid` | Detail popup opened from a CC item | `CCChildPageModel` |
| `node.config.taskCardUid` | Card content in the To-Do Center's CC list | `CCTaskCardDetailsModel` |

When configuring these fields through an agent or API-assisted workflow:

1. Read the CC node first and keep an existing `ccUid` or `taskCardUid`; do not replace an already bound UI root unless the user explicitly requests a replacement.
2. For first-time setup, create the surface through the CC node's matching **User interface** or **Task card** configurator. That flow generates the UID, creates the expected root FlowModel tree, and persists the UID back to the node config.
3. Do not invent a UID, reuse a UID from another node, or call the approval-only `flowSurfaces applyApprovalBlueprint` route. A generated identifier without the CC FlowModel tree is an incomplete configuration.
4. For an existing bound root, use `flowSurfaces get` and `flowSurfaces catalog` before localized edits, then use only operations and block/field types exposed by that root's catalog.
5. After writing, re-read both sides of the binding: require the node config to contain the expected UID and require that UID to resolve to the expected root model with the intended non-empty content. A detached, missing, or wrong-model root is failed verification.

## Branch Description
Does not support branching.

## Test Support
Not supported. This node cannot use CLI `workflow flow-nodes test` or HTTP `flow_nodes:test`, because the server-side instruction does not implement `test()`.

## Example Configuration
```json
{
  "users": ["{{ $context.data.ownerId }}", { "filter": { "id": "{{ $context.data.managerId }}" } }, 123],
  "ccUid": "<persisted CCChildPageModel root uid>",
  "taskCardUid": "<persisted CCTaskCardDetailsModel root uid>",
  "title": "{{ $context.data.title }} - CC"
}
```

Replace the example placeholders with the actual UIDs persisted by the CC UI configurators. Never submit the placeholder strings as node configuration.

## Output Variables
This node does not output variables.
