# Case 4 - 项目与任务协同页面

## 测试目标

验证筛选区块、主表、详情弹窗、关联任务表和新增/编辑表单在同一个业务流里的协同能力。

## 前置数据模型

- `projects`: 项目，字段至少包含 `id`、`name`、`status`、`owner_id`、`start_date`、`end_date`
- `tasks`: 任务，字段至少包含 `id`、`project_id`、`assignee_id`、`title`、`status`、`priority`
- `users`: 用户，字段至少包含 `id`、`nickname`、`email`
- 关系：`tasks.belongsTo(projects)`
- 关系：`tasks.belongsTo(users)`，作为负责人

## 测试 Prompt

```text
请使用 nocobase-ui-builder skill 帮我搭建一个“项目协同工作台”，并记录完整工具调用日志，结束后自动生成复盘报告。

页面目标如下：
1. 创建 v2 页面，标题为“项目协同工作台”。
2. 页面顶部放项目筛选区块，可按项目名称、状态、负责人筛选。
3. 页面主体放项目表格，展示项目名称、状态、负责人、开始日期、结束日期。
4. 项目表格支持“查看项目”动作，打开项目详情抽屉。
5. 项目详情抽屉中展示项目详情区块和任务表格。
6. 任务表格支持新增任务和编辑任务。

执行要求：
- 不要只创建空壳页面，要尽可能把字段和动作配置完整。
- 如果某个子场景做不到，要保留已完成部分并明确记录阻塞点。
- 最终输出日志路径、报告路径和提速建议。
```

## 最低验收标准

- 主页面壳、筛选区块和项目表格能创建完成
- 能尝试并说明详情抽屉和任务表格的落地情况
- 若新增/编辑任务动作不能完整打通，也要说明卡住的层级
- 工具日志和复盘报告存在
- 探测与写入顺序符合 skill 约定
- 结果足够用于判断这个业务流的真实可用性

## 重点验证的 NocoBase 能力

- `FilterFormBlockModel`
- `TableBlockModel`
- `DetailsBlockModel`
- `CreateFormModel`
- `EditFormModel`
- `openView` 与 popup page 编排

## 当前预期

这是主干偏高级的流程型场景，应尽量完成主页面和至少一层详情弹窗。多层写入如果不稳定，应该能清晰暴露在哪个阶段开始失控。

## 复盘关注点

- 是否可以通过更高层的 block tree upsert 减少多次 mutate
- popup 中任务表格的关系上下文是否足够明确
- 筛选区块与主表之间是否需要更清晰的联动模板
