# Case 7 - 组织架构树页面

## 测试目标

故意验证树表、自关联关系和新增下级节点这类偏复杂的边界场景，评估当前 API 与 skill 是否真的能处理层级型业务数据。

## 前置数据模型

- `departments`: 部门，字段至少包含 `id`、`name`、`parent_id`、`manager_id`
- `users`: 用户，字段至少包含 `id`、`nickname`、`department_id`
- 关系：`departments.belongsTo(departments)`，通过 `parent_id` 自关联
- 关系：`users.belongsTo(departments)`

## 测试 Prompt

```text
请使用 nocobase-ui-builder skill 帮我创建一个“组织架构”页面，并记录完整工具日志，结束后生成复盘报告。

页面目标如下：
1. 创建部门树表页面，部门表格启用树表模式。
2. 表格展示部门名称、上级部门、负责人。
3. 为每条部门记录提供“新增下级部门”动作。
4. 为每条部门记录提供“编辑部门”动作。
5. 点击部门后可以查看该部门下的用户列表，用户列表可以放在详情区域、抽屉或其他你认为当前最稳的方式。
6. 如果自关联树表或新增下级动作无法稳定完成，请明确指出是树表、record action 还是关系区块能力不足。
```

## 最低验收标准

- 能明确验证树表模式是否可建
- 能对新增下级部门动作的可行性给出明确结论
- 能尝试并说明部门用户列表的挂载结果
- 工具日志与复盘报告成功生成
- 不把失败场景伪装为成功
- 输出里明确指出最值得补强的 API 或 skill 缺口

## 重点验证的 NocoBase 能力

- `TableBlockModel` 的 `treeTable` 相关设置
- 自关联关系数据
- `AddChildActionModel` 或等价记录动作
- 部门详情与部门用户关系表

## 当前预期

这是边界用例，预期会比前面的主干 case 更容易暴露问题。重点不是强求跑通，而是稳定找出树形数据场景中最阻碍可用性的点。

## 复盘关注点

- 树表是否缺少更直观的高层模板
- 新增下级部门动作是否过于依赖隐式上下文
- 自关联数据是否需要更明确的 relation-block 支持
