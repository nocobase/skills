# Case 8 - 项目成员多对多管理

## 测试目标

故意验证多对多关系、中间表字段和关联选择弹窗的组合场景，评估当前 skill 和 API 在复杂关系数据块上的真实能力边界。

## 前置数据模型

- `projects`: 项目，字段至少包含 `id`、`name`、`status`
- `users`: 用户，字段至少包含 `id`、`nickname`、`email`
- `project_members`: 项目成员中间表，字段至少包含 `id`、`project_id`、`user_id`、`role`、`joined_at`
- 关系：`projects.belongsToMany(users)`，通过 `project_members`
- 关系：`project_members.belongsTo(projects)`
- 关系：`project_members.belongsTo(users)`

## 前置模拟数据

- `projects` 至少 4 条，覆盖不同状态
- `users` 至少 6 条
- `project_members` 至少 8 条，覆盖 `owner`、`pm`、`developer`、`tester` 等角色
- 至少准备 1 个项目拥有 3 名以上成员，且成员角色不同
- 至少准备 1 条便于后续编辑角色的中间表样本，确保 `joined_at` 等中间表字段也有值

## 测试 Prompt

```text
请使用 nocobase-ui-builder skill 帮我创建一个“项目成员管理”页面，并记录完整工具日志，结束后生成复盘报告和改进建议。

页面目标如下：
1. 创建项目详情页面或项目主表 + 详情页的结构。
2. 在项目详情中展示项目成员关系表格。
3. 项目成员表格要能展示用户、角色、加入时间。
4. 提供“添加成员”动作，期望通过弹窗选择用户并写入中间表。
5. 提供“编辑成员角色”动作。
6. 如果当前多对多关系块、关联选择器或中间表编辑做不到，请明确指出具体阻塞点，不要省略失败信息。

执行要求：
- 开始搭建 UI 之前，先准备并校验前置模拟数据，不要跳过这一步直接验证空页面。
- 最终说明里要交代项目、用户、中间表样本数量，以及哪个项目被用作主验证样本。
```

## 最低验收标准

- 项目页面主体能建起来
- 项目成员关系表格至少能读到一个拥有多名成员的真实项目样本
- 项目成员关系表格的能力边界被清楚验证
- 对添加成员和编辑角色动作给出明确结论
- 工具日志和报告产物齐全
- 说明中能区分多对多关系、中间表字段、弹窗选择器分别卡在哪
- 最终结果可直接作为关系块优化需求输入

## 重点验证的 NocoBase 能力

- 多对多关系数据块
- 中间表字段展示
- 关联记录选择弹窗
- 关系记录编辑

## 当前预期

这是典型的边界 case，预期很可能无法完整跑通，但它能非常有效地暴露关系区块、关联选择和中间表编辑的短板。

## 复盘关注点

- 当前 API 是否缺少 relation block 的聚合表达
- 中间表字段是否需要更明确的 schema 模板
- skill 是否应该把多对多场景单独建成一套测试与提示词策略
